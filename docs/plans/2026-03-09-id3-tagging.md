# ID3 Tagging Backend

## Overview

Add audio metadata (ID3/MP4/Vorbis) reading to Harmonic. This enriches the track list with real title, artist, album, year, genre, and track number from embedded tags instead of relying on filenames. Album art is served to the frontend via an HTTP handler.

Read-only for v1. Writing tags is a separate concern deferred to a future plan.

---

## Library choice: `dhowden/tag`

```
go get github.com/dhowden/tag
```

### Why this library

- **Unified API**: A single `tag.ReadFrom(io.ReadSeeker)` call returns a `Metadata` interface with `.Title()`, `.Artist()`, `.Album()`, `.Year()`, `.Genre()`, `.Track()`, `.Disc()`, `.Picture()`, `.Lyrics()`, `.Composer()`, `.AlbumArtist()`, `.Comment()`.
- **Format coverage**: ID3v1, ID3v2 (2.2, 2.3, 2.4), MP4 atoms (m4a, aac, alac), FLAC Vorbis comments, OGG Vorbis comments. This covers the majority of Harmonic's supported formats.
- **Album art extraction**: `Metadata.Picture()` returns a `*tag.Picture` with `Ext` (file extension), `MIMEType`, and `Data` (raw bytes).
- **Lightweight**: Only reads tag headers, doesn't decode audio. Fast enough to call on every file during scan.
- **Zero dependencies**: Pure Go, no CGO.

### Alternatives considered

| Library | Verdict |
|---------|---------|
| `bogem/id3v2` | Read+write but MP3-only. Too narrow for a multi-format player. |
| `slotheroo/yurit` | Fork of dhowden/tag with minor additions. Prefer the well-maintained original. |
| FFmpeg/ffprobe via exec | Heavy dependency, CGO or subprocess. Overkill for tag reading. |

### Format coverage gaps

`dhowden/tag` does **not** support these formats Harmonic scans for:

| Format | Tag system | Fallback |
|--------|-----------|----------|
| WAV | No standard tags (sometimes RIFF INFO or ID3v2 chunk) | Filename-derived title |
| AIFF/AIF | Sometimes ID3v2 chunk | Filename-derived title |
| WV (WavPack) | APEv2 tags | Filename-derived title |
| WMA | ASF metadata | Filename-derived title |

Opus files in OGG containers **are** supported (OGG Vorbis comments path).

When `tag.ReadFrom` returns `tag.ErrNoTagsFound`, we fall back to the filename-derived title that `ScanDirectory` already produces. No error is surfaced to the user — a file without tags just has less metadata.

## Architecture decision: Where does tag reading happen?

### Option A: New `TagService` in `internal/tags/`

A separate Wails-bound service. Frontend calls `ReadTags(path)` on demand (e.g., when a track is selected or played).

- Scan stays fast (filenames only).
- Track list shows filename-derived titles until the user interacts with a track.
- Poor UX: no artist/album column in the list without extra loading.

### Option B: Enrich `AudioFile` during scan

`ScanDirectory` reads tags for each file as it walks the directory. Returns richer objects with real metadata.

- All data available immediately when the list renders.
- Scan gets slower.

### Option C: Two-phase — scan then enrich

`ScanDirectory` returns file paths quickly (as now). A second call `ReadAllTags(paths)` reads tags in parallel and returns enriched results. Frontend shows the list immediately with filenames, then swaps in real metadata when it arrives.

### Recommendation: Option B with parallelism

Reading tags is fast per-file — `dhowden/tag` only reads the tag header (first few KB), not the full audio stream. The cost is roughly:

- Open file + read header + parse: ~0.5ms per file
- 1,000 files sequentially: ~500ms
- 5,000 files sequentially: ~2.5s
- 5,000 files with 8 workers: ~300ms

Since `ScanDirectory` already walks the tree (which is the slow part — the `"Slow af"` comment in the code), adding tag reads inside the walk doesn't change the order of magnitude. Using a worker pool makes it fast enough that the user won't notice.

The implementation uses a pipeline: `WalkDir` feeds file paths into a channel, a pool of goroutines reads tags and builds `AudioFile` structs, results are collected and sorted.

## Step 1: New package — `internal/tags/`

### New file: `internal/tags/read.go`

This package wraps `dhowden/tag` and returns Harmonic-specific types. It does not depend on Wails or any service — it's a pure library.

#### `TrackMeta` struct

The enriched metadata type. JSON-tagged for Wails serialization:

```go
type TrackMeta struct {
    Title       string `json:"title"`
    Artist      string `json:"artist"`
    Album       string `json:"album"`
    AlbumArtist string `json:"album_artist"`
    Genre       string `json:"genre"`
    Year        int    `json:"year"`
    TrackNum    int    `json:"track_num"`
    TrackTotal  int    `json:"track_total"`
    DiscNum     int    `json:"disc_num"`
    DiscTotal   int    `json:"disc_total"`
    HasArt      bool   `json:"has_art"`
}
```

`HasArt` is a boolean so the frontend knows whether to attempt loading art (avoids a 404 request for every track without art).

#### `ReadFile(path string) (TrackMeta, error)`

Opens the file, calls `tag.ReadFrom`, maps the `Metadata` interface to a `TrackMeta` struct. If `tag.ReadFrom` returns `tag.ErrNoTagsFound`, return a zero-value `TrackMeta` with no error (not an error condition — just a file without tags).

```go
func ReadFile(path string) (TrackMeta, error) {
    f, err := os.Open(path)
    if err != nil {
        return TrackMeta{}, err
    }
    defer f.Close()

    m, err := tag.ReadFrom(f)
    if err != nil {
        if errors.Is(err, tag.ErrNoTagsFound) {
            return TrackMeta{}, nil
        }
        return TrackMeta{}, err
    }

    trackNum, trackTotal := m.Track()
    discNum, discTotal := m.Disc()

    return TrackMeta{
        Title:       m.Title(),
        Artist:      m.Artist(),
        Album:       m.Album(),
        AlbumArtist: m.AlbumArtist(),
        Genre:       m.Genre(),
        Year:        m.Year(),
        TrackNum:    trackNum,
        TrackTotal:  trackTotal,
        DiscNum:     discNum,
        DiscTotal:   discTotal,
        HasArt:      m.Picture() != nil,
    }, nil
}
```

#### `ReadArt(path string) (data []byte, mimeType string, err error)`

Opens the file, calls `tag.ReadFrom`, extracts the picture. Returns the raw image bytes and MIME type. Returns an error if no art is found (the HTTP handler translates this to 404).

```go
func ReadArt(path string) ([]byte, string, error) {
    f, err := os.Open(path)
    if err != nil {
        return nil, "", err
    }
    defer f.Close()

    m, err := tag.ReadFrom(f)
    if err != nil {
        return nil, "", err
    }

    pic := m.Picture()
    if pic == nil {
        return nil, "", errors.New("no art found")
    }

    return pic.Data, pic.MIMEType, nil
}
```

### New file: `internal/tags/read_test.go`

Test `ReadFile` with a few real audio files. Create a `testdata/` directory with small sample files (a few seconds of silence in mp3 and flac with known tags). Tests should verify:

- Known title/artist/album are returned for a tagged mp3.
- Known title/artist/album are returned for a tagged flac.
- A tagless WAV returns zero-value `TrackMeta` with no error.
- A nonexistent file returns an error.
- `ReadArt` returns image bytes for a file with embedded art.
- `ReadArt` returns an error for a file without art.

## Step 2: Enrich `AudioFile` and `ScanDirectory`

### Modify: `internal/fs/dir.go`

#### Expand the `AudioFile` struct

Add tag-derived fields:

```go
type AudioFile struct {
    Title       string `json:"title"`         // from tag, falls back to filename
    Artist      string `json:"artist"`
    Album       string `json:"album"`
    AlbumArtist string `json:"album_artist"`
    Genre       string `json:"genre"`
    Year        int    `json:"year"`
    TrackNum    int    `json:"track_num"`
    TrackTotal  int    `json:"track_total"`
    DiscNum     int    `json:"disc_num"`
    DiscTotal   int    `json:"disc_total"`
    HasArt      bool   `json:"has_art"`
    FilePath    string `json:"file_path"`
    Ext         string `json:"ext"`
}
```

The `Title` field uses the tag title if present, otherwise falls back to the filename-derived title (current behavior). This means the frontend never needs to handle the fallback — it always just reads `.title`.

#### Parallelize `ScanDirectory` with a worker pool

Replace the current inline approach with a two-phase pipeline:

**Phase 1: Walk.** `filepath.WalkDir` collects all matching file paths into a slice. This is the same logic as today, but instead of building `AudioFile` structs inline, it just collects paths + filename-derived titles.

**Phase 2: Read tags in parallel.** A pool of N goroutines (use `runtime.NumCPU()`, capped at say 8) reads from a channel of file entries, calls `tags.ReadFile()` for each, and sends enriched `AudioFile` structs to a results channel.

Sketch of the pipeline:

```go
func (s *FsService) ScanDirectory(dir string) ([]AudioFile, error) {
    // Phase 1: collect file paths
    type entry struct {
        path         string
        filenameName string // title derived from filename
        ext          string
    }
    var entries []entry
    err := filepath.WalkDir(dir, func(path string, d os.DirEntry, err error) error {
        // ... same extension filtering as now ...
        // append to entries instead of building AudioFile
    })

    // Phase 2: read tags in parallel
    workers := runtime.NumCPU()
    if workers > 8 {
        workers = 8
    }

    results := make([]AudioFile, len(entries))
    var wg sync.WaitGroup
    ch := make(chan int, len(entries))
    for i := range entries {
        ch <- i
    }
    close(ch)

    for w := 0; w < workers; w++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for i := range ch {
                e := entries[i]
                meta, _ := tags.ReadFile(e.path) // errors are silently ignored — just less metadata
                af := AudioFile{
                    FilePath:    e.path,
                    Ext:         e.ext,
                    Title:       meta.Title,
                    Artist:      meta.Artist,
                    Album:       meta.Album,
                    AlbumArtist: meta.AlbumArtist,
                    Genre:       meta.Genre,
                    Year:        meta.Year,
                    TrackNum:    meta.TrackNum,
                    TrackTotal:  meta.TrackTotal,
                    DiscNum:     meta.DiscNum,
                    DiscTotal:   meta.DiscTotal,
                    HasArt:      meta.HasArt,
                }
                // Fall back to filename if tag title is empty
                if af.Title == "" {
                    af.Title = e.filenameName
                }
                results[i] = af
            }
        }()
    }
    wg.Wait()

    return results, err
}
```

Key details:
- **Results are index-addressed** (`results[i]`), not appended, so no mutex is needed. Each goroutine writes to a distinct index.
- **Tag read errors are silently dropped** — if a file can't be parsed, the `AudioFile` just has empty metadata fields and the filename-derived title.
- **Sort order is preserved** — `results` has the same order as `entries`, which is `filepath.WalkDir` order (lexicographic within each directory).

## Step 3: Album art HTTP handler

### New file: `internal/tags/handler.go`

An `http.Handler` that serves embedded album art for a given file path. Same pattern as the audio file handler from the audio player plan.

```go
func NewArtHandler() http.Handler
```

The handler:

1. Only accepts `GET` requests. Return `405` otherwise.
2. Reads the `path` query parameter: e.g. `/art?path=/home/user/music/song.mp3`.
3. Validates the file extension against the audio extensions set.
4. Calls `ReadArt(path)` to extract the embedded image.
5. If no art found, return `404`.
6. Sets `Content-Type` to the MIME type from the picture (typically `image/jpeg` or `image/png`).
7. Sets `Cache-Control: public, max-age=86400` — album art for a given file won't change during a session, so aggressive caching is fine. This means the browser only fetches each cover once.
8. Writes the image bytes to the response.

### Wire into `main.go`

The Wails `AssetServer.Handler` only takes a single `http.Handler`. Since the audio player plan also adds a handler (`/audio?path=...`), you need a small mux that dispatches by path prefix.

Two approaches:

**Option A: `http.ServeMux`**

```go
mux := http.NewServeMux()
mux.Handle("/audio", fs.NewAudioHandler())
mux.Handle("/art", tags.NewArtHandler())

// ...
AssetServer: &assetserver.Options{
    Assets:  assets,
    Handler: mux,
},
```

**Option B: Single handler that switches on `r.URL.Path`**

Either works. `http.ServeMux` is cleaner and more extensible. Use that.

### Frontend usage

In any component that needs to show album art:

```svelte
{#if track.has_art}
  <img src="/art?path={encodeURIComponent(track.file_path)}" alt="" />
{:else}
  <div class="art-placeholder">♪</div>
{/if}
```

The `has_art` boolean from `AudioFile` avoids unnecessary 404 requests for tracks without embedded art.

## Future: Write support

Not in this plan. Writing tags is complex:

- Different libraries per format (`bogem/id3v2` for MP3, custom code for MP4/FLAC).
- Writing can corrupt files if padding/offsets are wrong.
- Need a UI for editing fields (inline editing or a modal editor).
- Need to handle the "save changed tags?" UX flow.

When the time comes, scope it as a separate plan. Read support is self-contained and useful on its own.
